## 1. Is my plan for grouping health conditions okay? (most important
question)

**Background:** One of the two files I'm working with is a medical
history file — it lists health conditions for each patient as free-text
entries (plain written phrases like "COPD/Emphysema" or "Type 2
Diabetes"), not official medical billing codes. My goal is to turn this
into a measure of each patient's overall health condition burden
("comorbidity burden"), so I can check whether adjusting for it changes
the original finding about SSRI use and survival.

The standard tool for this in the research literature is the Charlson
Comorbidity Index (CCI) — a scoring system that assigns point values to
17 different chronic conditions (heart disease, kidney disease, liver
disease, cancer history, and so on) and adds them up into one score per
patient. Two of the reference papers use this: Paper 2 calculates the
full 17-category CCI score and splits patients into "high" and "low"
groups to see if that predicts survival. Paper 1 takes a narrower
approach — instead of all 17 categories, they focus on just the four
most common ones found in their patient population: cardiovascular
disease (heart and blood vessel problems), chronic obstructive pulmonary
disease or COPD (long-term lung damage), type 2 diabetes, and rheumatic
and musculoskeletal disorders (joint and connective tissue conditions
like arthritis).

**What I'm asking:** I'm planning to follow Paper 1's narrower,
four-category approach rather than the full CCI, because my data is
plain free text without medical codes attached — reliably sorting
entries into all 17 CCI categories would require clinical judgment calls
I'm not equipped to make confidently from text alone. The four-category
approach uses broader, more obviously-identifiable conditions. Does this
tradeoff sound reasonable to you, or would you rather I attempt the
fuller CCI despite the added difficulty, or use some other set of
categories entirely?

**Why this matters:** This decision shapes every step that comes after
it — how I sort the medical history entries, what the resulting patient
comorbidity data looks like, and what goes into the final survival
model. If I start sorting under one plan and need to switch later, that
work has to be redone.

## 2. Are the extra patient ID numbers in the medical history file
expected?

**Background:** I have two data files that are both supposed to
describe the same 1,301 patients enrolled in the S1404 trial: the main
trial file (with treatment arm, age, survival outcomes, etc.) and the
medical history file (with each patient's listed health conditions).
Each patient has a unique ID number in both files, which I use locally
in R to eventually connect the two files together — that ID number
never leaves my computer or gets shared with any outside tool.

**What I found:** When I counted unique patient ID numbers in each
file, the medical history file has 142 ID numbers that don't match any
of the 1,301 patients in the main trial file. I checked carefully to

slightly differently in one file versus the other — and that wasn't the
cause; the ID numbers themselves are just genuinely different people not
present in the main file.

**What I'm asking:** My best guess is that the medical history file may
include people who were screened as candidates for the trial but never
actually got randomized into it, meaning they'd have medical history
recorded but no trial outcome data. Does that sound like a plausible
explanation given how S1404's data collection worked? And is it correct
that I should simply exclude these 142 non-matching people from my
analysis, since they were never part of the actual 1,301-patient trial
population?

**Why this matters:** When two files that are supposed to describe the
same group of people don't fully agree with each other, it's worth
confirming the reason before trusting downstream numbers built from
them.

excluded, rather than accidentally missed?

**Why this matters:** This affects how I build the classification step,
and how easy it will be for anyone (including you) to later audit or

**Why this matters:** This is a genuine assumption I'm making without
