# Questions for Mentor — 2026-07-01

Open items from today's session, in priority order.

## 1. Comorbidity framework confirmation (highest priority)

Leaning toward Paper 1's 4-category approach (CVD, COPD, T2D, RMD)
instead of the full Charlson Comorbidity Index, because our medical
history data is free-text without ICD codes — sorting into all 17 CCI
categories would require clinical judgment calls that aren't reliably
possible from plain-language terms alone. Does this tradeoff seem right,
or is there a reason to push for the fuller CCI despite the added
classification burden?

## 2. Non-randomized patients in the medical history file

Found 124 PATNO values in the medical history file that don't match any
of the 1,301 trial patients in the main dataset — well-formed IDs, same
format as everyone else, just not part of the analysis population.
Best guess: medical history collected at screening, before
randomization, for patients who were screened but not ultimately
enrolled. Can you confirm whether this is expected, and whether it's
safe to simply exclude these (which happens automatically once the
comorbidity matrix is joined onto the main dataset)?

## 3. Unusual-format patient IDs

18 rows in the medical history file have a PATNO value in an unusual
18-character format, different from the standard 6-character IDs
everywhere else. Any idea what this format might represent? Low impact
either way (18 rows out of 8,991), but wanted to flag rather than
silently drop.

## 4. Handling non-clinical MHTERM entries

The medical history file includes entries that are clearly not
comorbidities in the clinical sense — prior surgeries, allergies, and
similar. Should these be excluded entirely during classification, or
kept in a separate "other / non-comorbidity" bucket for transparency?

## 5. MHONGO handling for ambiguous entries

17 medical history entries have a real MHTERM value but no MHONGO
(ongoing/not-ongoing) flag recorded. Current working assumption: treat
these the same as MHONGO = 1 (count toward comorbidity burden), since
ongoing status can't be ruled out. Does this seem like a reasonable
default, or is there a better way to handle it?
